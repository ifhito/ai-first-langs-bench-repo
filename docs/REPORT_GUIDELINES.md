# REPORT.md 執筆規約（全 problem 共通）

各 problem の `REPORT.md` を、**結論を後から誰でも検証できる**形に揃えるための書式規約。
problem1 / problem2 とも、また再実行時もこの規約に従って書くこと。
各ルールには「なぜ必要か（＝何の検証のためか）」を併記する。
（具体例は problem1＝JSONパーサのものを引いているが、ルール自体は課題非依存。`22/22` は「全テストケース通過」と読み替える。）

---

## A. 検証可能性（claim は必ず裏が取れる形で書く）

1. **すべての数値主張に再実行コマンドを併記する。**
   「全ケース通過」等の主張は、（Docker等の固定環境で）実行できる具体的コマンドを表に載せ、レポート作成側でも**独立に再実行して確認**する（＝採点の二重化）。
   - *なぜ*：エージェントの自己申告を鵜呑みにしないため。第三者が同じコマンドで追試できる。
   - 例（problem1）：`zero run jsonparser.0` → `PASS 22/22` を作成側でも再実行して確認。

2. **数値は `results/{言語}/result.json` と完全に一致させる。**
   表とJSONで食い違う場合はJSONを正とし、差異は注記する。
   - *なぜ*：一次データ（JSON）と要約（表）の乖離を防ぐ。

3. **冒頭に必ず：使用モデル名・実行日・環境（処理系をバージョン固定で列挙）を書く。**
   - *なぜ*：結果はモデル・日付・処理系バージョンに依存する。再現の前提条件を固定する。

---

## B. 公平な比較（A/B・言語間を同条件に保つ）

4. **「文法ガイド なし＝パターンA／あり＝パターンB」を定義してから使う。**
   A は仕様のみ（言語ガイド禁止・ツール操作の確認のみ可）、B は公式ガイド供給。訓練データが豊富な対照群（例：JavaScript / Python）は A のみ。
   - *なぜ*：A/B の差分を「ガイドの効果」と言うには、A でガイドを使っていないことが保証されている必要がある。

5. **「なぜ高い/低いか」の因果を書くときは、判別シグナルを必ず添える。**
   高スコアの原因を「①モデルが元から知っていた」と「②設計（ガイド/エラー）で通った」のどちらに帰属するかは、次の観測値で裏づける：
   - **A の「1回でpass」が ✅** → ①知っていた寄り（例：MoonBit）。
   - **A の「1回でpass」が ❌ なのに最終 全通過** → ②設計で通った寄り（例：NanoLang は A で事前知識ほぼ全滅→エラーから文法を逆算して到達）。
   - **A=失敗 / B=成功の落差** → ガイドの効果（例：Zero・Vera は 0/22→22/22）。
   - *なぜ*：「高い理由が言語ごとに逆」といった主張を、印象でなく表の値（1回でpass・修正回数・A/B差）から検証できるようにするため。

6. **コスト指標（最終行数・トークン数・修正回数）も併記する。**
   pass率が同じでも「楽に通った」か「冗長な回避策で通った」かを区別する（例：NanoLang/A は 22/22 だが最長906行・4回修正）。
   - *なぜ*：pass率だけ見ると「設計で苦労して通した」ケースを過大評価する。

---

## C. 用語と表記の統一

7. **専門用語は日常語＋括弧で原語、を徹底する。**
   例：「とても親切（＝structured）」「期待/実際（expected/actual）」「`__ANON_0`（＝コンパイラ内部の符号名で、実際に書くべき記号が分からない）」。初出箇所か用語集で必ず定義する。
   - *なぜ*：読み手が前提知識なしに結論を検証できるように。

8. **ラベルは固定**：`文法ガイド`（なし/あり）・`1回でpass`・`修正回数`・`最終`・`最終行数`・`トークン数`・`自己解決`・`エラーの質`。
   - 英語フィールド名（`pass_at_1` / `fix_rounds` / `loc` / `tokens` / `self_recovered` 等）は result.json 側で使い、レポート本文の表は上記の日本語ラベルに統一する。

9. **「エラーの質」は ◎/○/△ の3段階＋「具体的に何を教えたか」を一言で。**
   ◎とても親切（どの行で・何が・どう直すかまで分かる）／○ふつう（場所と原因は分かるが直し方は無い）／△役に立たない（エラーが出ない or 要点が伏せられている）。
   - *なぜ*：「親切な体裁か」より「足りない知識を指せたか」を見るための軸を固定する。

---

## D. 分類の厳密さと正直さ

10. **status の定義を厳守**：`DONE`（最終的に全ケース通過）／`PARTIAL`（規定回数の修正でも未達）／`SKIPPED`（**処理系が存在しない場合のみ**）。
    処理系は動くがコード生成に失敗したものは `PARTIAL`。誤分類は**再分類し、生の自己申告値と理由を残す**（例：Vera/A の SKIPPED→PARTIAL）。

11. **正規化したら必ず元の値も残す**（例：`self_recovered` を「人間介入なしに全通過したか」で正規化した場合、各エージェントの生値も result.json に併記）。

12. **Caveats を必ず書く**：手書き縛り・n=1のブレ・ツールチェーンのドリフト・再現範囲など、結論を弱める事実を隠さない。
    - *なぜ*：検証する人が「どこまで信じてよいか」を判断できるようにする。

---

## REPORT.md に含める内容（章立ての目安）

1. **概要**：モデル名・実行日・環境・何条件中いくつ実行できたか
2. **用語の説明**：パターンA/B、attempt と修正ループ、各指標（ルールC）
3. **検証（採点の二重化）**：全通過主張の再実行コマンドと結果（ルールA-1）
4. **結果サマリー表**：固定ラベル（ルールB-6, C-8, C-9）
5. **言語ごとの所感**：各2〜4行
6. **考察**：pass@1 の差の由来、既存言語 vs 新言語、親切なエラーの効果（因果は判別シグナル付き＝ルールB-5）
7. **Caveats**（ルールD-12）
8. 末尾に本規約への参照リンク
