## 用語の説明

### 生成条件：パターンA / パターンB
同じ仕様・同じ22ケースで、**与える材料だけを変えて**比較する。

| 条件 | 与えるもの | 測りたいこと |
| --- | --- | --- |
| **パターンA（仕様のみ）** | 仕様＋①処理系の起動方法②**自分の生成コードを動かして出た生エラー**のみ。文法ガイド・`zero explain` 等の解説/例示・stdlib解説・サンプルは**禁止** | モデルが**その言語を訓練データからどれだけ知っているか** |
| **パターンB（ガイド付き）** | 仕様に加えて、**公式の文法ガイド本文**（`zero skills get`／`/opt/vera/SKILL.md`／NanoLang同梱docs 等） | **設計介入（ガイド供給）で訓練データ不足を補えるか** |

- JavaScript / Python は対照群のため**パターンAのみ**。新4言語は A・B 両方を `results/{言語}/result.json` の `conditions.A`/`.B` に分けて記録。

### attempt（試行）と修正ループ
失敗したら**処理系の生エラーだけ**を根拠に直し再実行する。各回が **attempt**。
- `attempt_1` = 最初の生成。`attempt_2` … = 各修正後。
- **修正は最大5回まで（＝最大 `attempt_6`）。`attempt_6` でも未達なら打ち切り `PARTIAL`。規定外（6回以上）で通っても採用しない。**
- **`fix_rounds` ＝「全通過に到達した attempt 番号 − 1」**（pass@1=0、attempt_3到達=2）。未達は5。最終attempt番号と必ず一致させる。

### 「値で一致」と「正規形で一致」（`pass_basis`）
同じ「22/22」でも**何を比較したか**が言語で違うため区別する。
- **`value`（値比較）**：パース結果を**ネイティブな値**として期待値と比較（JS/Python の深い等価、MoonBit の ADT `derive(Eq)` 等）。
- **`normalized`（正規形比較）**：値ADTを実行時に構築できない言語が、**正準化した文字列/バイト列**を期待の正準形と比較（Zero/B・NanoLang/B・Vera/B）。「値そのものの比較」より一段弱い同一性であり、**別カテゴリ**として読む。

### その他の指標
- **pass@1**：最初の生成だけで22/22か。
- **status**：`DONE`（22/22）／`PARTIAL`（5回でも未達）／`SKIPPED`（処理系が無い）。
- **self_recovered**：人間の介入なしに22/22へ到達したか（pass@1も✅に正規化。生値は result.json 併記）。
- **loc / tokens**：最終版の行数／概算トークン数。
- **error_quality（エラーの質）**：◎ とても親切／○ ふつう／△ 役に立たない（表直後に定義）。

## 言語ごとの所感

### JavaScript（A）
仕様だけで手書き再帰下降パーサを一発生成、22/22（値比較）。境界条件（`01`/`1.`/`.5`、末尾カンマ、閉じ忘れ、トリム、エスケープ）も初回通過。対照群として機能。

### Python（A）
同じく手書き再帰下降＋独自例外で一発 22/22（値比較）。JSより少ないトークンで同等のロジック。エラーが修正に効く場面は無し。

### Zero（A→22/22・修正4回、B→22/22・修正1回）
両条件とも**「型はあるのに実行エンジンが対応しない」制約**が支配的。値ADT・record・文字列連結等が実行用バックエンドで使えず、parser は整数バッファ＋位置渡しに退避。**A（ガイドも `zero explain` も禁止）でも、自分のコードに対する生エラーが具体的で、文法を逆算して 22/22 に到達**（値比較。スカラは値で、構造は妥当性で判定）。B はガイドで設計を素早く確定し、明示スタック＋**正規化シリアライズのバイト比較**で 22/22（`normalized`）。

### MoonBit（A→22/22 pass@1、B→22/22 pass@1）
**A・B とも一発 22/22**（値比較）。ML系で成熟しモデルが熟知。ADT `enum Json … derive(Eq)` で値の構造等価をそのまま使える。B では公式 `llms.txt` が不達のためツールチェーン自体（足場・`moon test` 診断）をガイド代用したが、論理修正は不要だった。

### NanoLang（A→22/22・修正4回、B→22/22 pass@1）
**構造化（Elmスタイル）エラーの威力を最も鮮烈に示した**。A では事前知識がほぼ外れたが、「呼び出しは `(f 引数)`」「代入は `set`」等、**正しい書き方を直接提示するヒント**だけからS式文法を逆算し4回で 22/22（値比較）。B は同梱docsを読み一発だが、struct フィールド代入不可のため**正準文字列エンコードでの比較**に倒した（`normalized`）。

### Vera（A→0/22 PARTIAL、B→22/22・修正1回）
**A は5回直しても本物のパーサを書けず 0/22**。位置引数や契約の構文はエラーから逆算できたが、**引数名を束縛するトークンがエラー上 `__ANON_0` と不透明**で、入力文字列を参照する関数が書けなかった（attempt_6 はコンパイル/実行できるが定数 Err スタブ＝機能的に 0/22）。B では同梱 `SKILL.md` を読み、契約と再帰の `decreases()`/`effects(pure)` を理解して 22/22。比較は**再シリアライズした正準形の文字列比較**（`normalized`）。

## 考察

### pass@1 の差はどこから来るか（判別シグナル付き＝規約B-5、ただし n=1 のため断定はしない）
- **この1試行では、pass@1 の差はモデルの“言語既知度”と整合する**。JS/Python と **MoonBit/A** は pass@1（→「①元から知っていた」寄り）。
- **A=失敗だが最終全通過**の **Zero/A・NanoLang/A** は「②親切なエラーから逆算して通った」寄り（pass@1 ❌ → 生エラー駆動で 22/22）。ガイド禁止でも、エラーが具体的なら到達できた。
- **A=失敗のまま**は **Vera/A のみ**（0/22）。引数構文が内部記号で隠れ、生エラーから復元できなかった。

### 既存言語 vs 新言語 ＝ ガイド（設計介入）の効果
- **ガイドが決定的に効いたのは Vera**（A=0/22 → B=22/22）。一関数も書けない状態から、ガイドで到達できた。
- 一方 **Zero/NanoLang は A（ガイド無し）でも 22/22 に到達**しており、「新言語＝Aで失敗」という単純構図ではない。ガイドの主効果は **Vera の救済**と、**他言語での修正回数・コスト削減**（例：Vera/B は5回前提が1回、MoonBit/B は一発）に表れた。
- ただし **`normalized` で通った3条件（Zero/B・NanoLang/B・Vera/B）は「値そのもの」ではなく正準形の比較**で達成しており、`value` 条件と同列に「22/22」を並べるときは基盤の違いに注意。

### 親切なエラーは修正ループで有利だったか
- **強く支持された**。最良例は **Zero/A・NanoLang/A**：公式ガイド無しでも、**正しい書き方を指す具体的なエラーだけ**から未知の文法を組み上げ 22/22 に到達。
- **限界**：Vera/A は体裁は親切でも、肝心の引数束縛子が内部記号 `__ANON_0` で隠れて復元できず 0/22。**「見た目の親切さ」より「足りない知識をズバリ指せるか」**が効く。

## Caveats（正直な注意書き）

- **手書き・標準JSON禁止はルールとして明文化済み**：`JSON.parse`/`json.loads`/`eval` 等の禁止と手書き必須を、各プロンプトの制約と `AGENT_INSTRUCTIONS.md` の固定ルールに記載した（以前は実行側の口頭指示だった点を是正）。
- **「22/22」の意味が一様でない（pass_basis）**：Zero/B・NanoLang/B・Vera/B は**正準化シリアライズの比較**で合格しており、値そのものの比較（JS/Python/MoonBit/Zero-A/NanoLang-A）とは別カテゴリ。各 `result.json` の `pass_basis` と結果サマリー直後の注で区別している。
- **試行回数は各条件1回（n=1）**：別シードで pass@1・修正回数・到達は変動しうる。特に「A で 22/22 に届くか」（Zero/A・NanoLang/A）と Vera/A の 0/22 は再現性に幅が出やすい。**結論は参考観測であり断定ではない。**
- **Vera/A は5回の修正を使い切っても 0/22**：引数名を束縛するトークンがエラー上 `__ANON_0`（内部記号）でしか現れず、ガイド無しでは関数の引数すら書けない（入力文字列を参照できないため本物のパーサにならない）。最大5回の修正を尽くした上での 0/22（PARTIAL, fix_rounds=5）。
- **Zero の特殊事情**：`zero check`（型検査）は通るのに実行用バックエンドが多くの構文を弾く（record/値ADT/文字列連結など）。本ベンチは**実行して22ケースを通すこと**を基準にしたため、実行可能な設計（整数バッファ＋正規化）を最終とした。Zero は codegen が遅い（数分）。
- **ガイド取得のばらつき**：MoonBit は公式配布ドキュメントが不達でツールチェーンをガイド代用。Zero のガイド取得コマンドのIDは `language`/`diagnostics`/`stdlib`（接頭辞付き名は不可）。NanoLang は同梱docsが一次情報。
- **採点ハーネスは各エージェント自作**：22ケースの期待値は `testcases.md` を唯一の正として固定し、22/22 主張は全てレポート側で独立再実行して確認済み（上表）。
- **再現範囲**：Docker で再現できるのは「生成済みコード × 22ケースの採点」。コード**生成**自体は AI エージェント駆動でコンテナ外（`AGENT_INSTRUCTIONS.md` 準拠）。詳細は `DOCKER.md`。
