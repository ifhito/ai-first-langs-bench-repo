## 用語の説明

このレポートと `results/{言語名}/result.json` で使う用語の定義。

### 生成条件：パターンA / パターンB
モデルに「どれだけの情報を与えて生成させたか」の2条件。同じ仕様・同じ22ケースで、**与える材料だけを変えて**比較する。

| 条件 | 与えるもの | 測りたいこと |
| --- | --- | --- |
| **パターンA（仕様のみ）** | JSONパーサの仕様だけ。言語の文法ガイドは**与えない**（処理系の呼び出し方などツール操作の確認は可） | モデルが**その言語を訓練データからどれだけ知っているか**（=素の言語知識） |
| **パターンB（ガイド付き）** | 仕様に加えて、**公式の文法ガイド**（`zero skills get`／`/opt/vera/SKILL.md`／NanoLang同梱README 等）を取得して与える | **設計介入（ガイド供給）で訓練データ不足を補えるか** |

- JavaScript / Python は訓練データが豊富な対照群のため**パターンAのみ**。
- Zero / MoonBit / NanoLang / Vera は**A・B両方**を実施し、`results/{言語}/result.json` の `conditions.A` / `conditions.B` に別々に記録している。
- 例：Zero は A=0/22 → B=22/22。「ガイドを足すと書けるようになった」差分がそのまま B の効果。

### attempt（試行）と修正ループ
1回の生成で全ケース通るとは限らない。失敗したら**処理系の生エラーだけ**を根拠に直し、再実行する。この各回が **attempt**。

- `attempt_1.<ext>` = 最初の生成。`attempt_2`, `attempt_3` … = 各修正後のコード（ファイルとして全部残す）。
- 各 attempt について「22ケース中いくつ通ったか／どのケースが落ちたか」を `result.json` の `attempts[]` に記録。
- **修正は最大5回まで**（`fix_rounds` が修正回数）。5回で全通過しなければ、その時点で打ち切り `PARTIAL` とする。
- 修正時は人間向けの解説を足さず、**生エラーを見て直す**（その言語のエラー出力の質を測るため）。

例（Zero/B の `attempts`）：`attempt_1〜7` が `0/22`、`attempt_8` で `22/22` → `fix_rounds` は8回目で到達。

### その他の指標
- **pass@1**：最初の生成（attempt_1）だけで全22ケース通ったか（true/false）。`true` なら修正ループ不要。
- **final_pass_count**：最終的に通ったケース数（`N/22`）。
- **status**：`DONE`（最終的に22/22）／`PARTIAL`（5回でも全通過せず）／`SKIPPED`（処理系がそもそも無い）。
- **self_recovered**：人間の介入なしに自力で 22/22 へ到達したか。本レポートでは「pass@1 で通った場合も到達＝true」に正規化（result.json には各エージェントの生値も併記）。
- **loc / tokens**：最終版コードの行数／概算トークン数（単語数）。
- **error_quality（エラーの質）**：処理系のエラーが間違いを直すヒントとしてどれだけ親切か。**◎ とても親切**（どの行で・何が・どう直すかまで分かる）／**○ ふつう**（場所と原因は分かるが直し方までは無い）／**△ 役に立たない**（エラーが出ない or 要点が伏せられている）の3段階。詳細は「結果サマリー」直後の『「エラーの質」の見方』を参照。

### 「エラーの質」の見方

「エラーの質」とは、**処理系が出すエラーメッセージが、間違いを直すヒントとしてどれだけ親切か**のこと。
このベンチでは、失敗したコードを人間の助言なしに「処理系が出した生のエラーだけ」を見て直させる。だから**エラーが親切なほど直しやすく、不親切だと直せない**。親切さを3段階で見る：

| 親切さ | どういうエラーか | 直しやすさ |
| --- | --- | --- |
| **◎ とても親切**（＝structured） | 「**どの行で・何が・本来どうあるべきで・どう直せばいいか**」まで具体的に書いてある | エラーを読むだけでほぼ直せる |
| **○ ふつう**（＝free） | ふつうの文章で書いてあり、場所と原因は分かるが、直し方までは教えてくれない | 自分で考えれば直せる |
| **△ 役に立たない**（＝poor） | そもそもエラーが出ない（一発で成功）か、出ても肝心な所が伏せられていて手がかりにならない | 直しようがない／そもそも直す必要なし |

**補足：よく出てくる言い回し**
- **「期待 / 実際」（expected/actual）**：エラーが「**本来ここに来るべきもの（期待）**」と「**実際にコードに書かれていたもの（実際）**」を並べて見せてくれること。
  例：『ここは `,`（カンマ）が来るはずなのに、実際は `(` が来ている』。これがあると、何をどう直せばいいか一目で分かる。

**いちばん大事な発見：「親切な体裁か」より「足りない知識をズバリ指し示せるか」が効く。**
- **良い例＝NanoLang（ガイドなし）**：エラーがあまりに具体的で、ガイドが無くても**エラーを読むだけで正しい書き方を逆算できた**（エラーがガイドの代わりになった）。
- **悪い例＝Vera（ガイドなし）**：体裁は親切でも、肝心の「何を書けばいいか」が内部記号（`__ANON_0`）で隠れていて分からず、**関数を1つも書けずに 0/22**。

（表のいちばん右の列「エラーの質」は、その言語のエラーが**具体的に何を教えてくれたか**を一言でまとめたもの。）

## 言語ごとの所感

### JavaScript（A）
仕様だけで手書き再帰下降パーサを一発生成、22/22。境界条件（`01`/`1.`/`.5`、末尾カンマ、閉じ忘れ、トリム、エスケープ）も初回通過。訓練データが豊富な典型課題で、新言語の対照群として機能。

### Python（A）
同じく手書き再帰下降＋独自例外で 22/22 を一発。JSより少ないトークン（523）で同等のロジックを表現。エラー出力が修正に効く場面は発生せず。

### Zero（A→0/22、B→22/22）
**ガイドの有無で天と地**。Aでは `zero check --json` の構造化診断（`PAR100`/`TYP021`/`ERR002` 等、expected/actual 付き）が表層の構文修正には効いたが、「Stringのバイト走査」「可変長コレクション」「sum型のペイロード分解」「rescue構文」といった**イディオムは事前知識から復元できず 0/22**。Bでは公式 `zero skills get` を読み、再帰を避けて明示スタックで正規化シリアライズする設計に切り替え、8回の修正（構造化エラー駆動）で 22/22 に到達。

### MoonBit（A→22/22 pass@1、B→22/22）
**新4言語で唯一、Aでも一発 22/22**。ML系で最も成熟しており、モデルが訓練データから既に熟知。Bでは2026年ビルドのツールチェーン・ドリフト（`?`後置演算子の廃止、`parse_int`系のraise化）に遭遇したが、`moon check` が**置換イディオムまで提示**し2回で解消。

### NanoLang（A→22/22、B→22/22 pass@1）
**構造化エラーの威力を最も鮮烈に示した**。Aでは事前知識がほぼ全滅だったが、Elmスタイルのエラー（名前付きコード＋ソースカレット＋「関数呼び出しは (f args) で」等の具体的ヒント）だけから**S式呼び出し/中置演算子の実文法を逆算**し、4回の修正で 22/22（906行）。Bでは同梱README/QUICK_REFERENCE/STDLIBを読んで一発 22/22（522行）。`shadow` テストブロック必須・`set s.field`非対応などの癖はガイドで把握。

### Vera（A→0/22、B→22/22）
**Aは一関数すら構文的に成立せず**（パラメータ宣言の `@`/De Bruijn 構文を事前知識から復元できず、肝心のトークンがエラー上 `__ANON_0` と不透明で手掛かりにならなかった）。Bでは同梱 `/opt/vera/SKILL.md` を読み、De Bruijn インデックス・`requires/ensures/effects` 契約・型付き効果を理解。バグの大半は De Bruijn スロット順序で、自然文診断と `vera check --explain-slots` のスロット表が決定打となり5回で 22/22。

## 考察

### pass@1 の差はどこから来るか
- **訓練データ量がほぼ全てを決めた**。JS/Python（潤沢）は pass@1 即 22/22。新言語のパターンAは、MoonBit（ML系で成熟・既知）だけが pass@1、Zero/NanoLang/Vera は失敗（Zero・Vera は 0/22 まで悪化）。
- **「言語の新しさ」より「モデルが知っているか」が効く**：MoonBit は新言語でも訓練データ上の露出が多く、JS/Python と同じ挙動を見せた。

### 既存言語 vs 新言語 ＝ ガイド（設計介入）で埋まるか
- **明確に埋まる**。パターンBで4新言語すべてが 22/22 に到達：
  - Zero: 0/22 → **22/22**
  - Vera: 0/22（一関数も書けない）→ **22/22**
  - NanoLang: 4回修正 → **pass@1**
  - MoonBit: 2回修正（ツールチェーン・ドリフトのみ）
- 「公式ガイドを与える」という設計介入は、**訓練データ不足を実際に補える**ことが示された。コスト（LOC/トークン）はBの方が大きい（正規化シリアライズ等の防御的設計に倒れるため）。

### 親切なエラーは修正ループで有利だったか
- **強く支持された**。最良の実例は **NanoLang（ガイドなし）**：ガイド禁止の条件下で、親切なエラー（間違いの場所を `^` で指し示し、「本来こう書く」という具体ヒント付き）**だけ**から言語の書き方を逆算し 22/22 に到達。エラー出力自体がガイドの代わりになった。
- Zero の「本来こう来るべき vs 実際こう来た」を並べる表示、MoonBit の「その記法は廃止。代わりにこう書く」という名指し提示、Vera の専用確認コマンド（`vera check --explain-slots`：引数の並び順を表で見せてくれる）も、いずれも修正を直接後押しした。
- 限界も観測：Vera（ガイドなし）は体裁は親切でも肝心の所が内部記号 `__ANON_0`（＝コンパイラ内部の符号名で、実際に書くべき記号が分からない）で隠れて役立たず、Zero（ガイドなし）は細かい書き方は直せても言語独特の書き方までは導けなかった。**「見た目が整っているか」より「足りない知識をズバリ指し示せるか」**が効く。

## Caveats（正直な注意書き）

- **標準ライブラリJSONの不使用は実行側の判断**：パターンAのプロンプト文面は `JSON.parse`/`json.loads` 等を明文禁止していないが、本ベンチの目的は「生成パーサコードの質」の測定であり、stdlibを使えば JS/Python が自明に通過し対照群として無意味になる。全エージェントに**手書き実装を明示指示**した。
- **試行回数は各条件1回（n=1）**：LLM生成にはブレがあり、別シードでは pass@1・修正回数・最終到達が変動しうる。特に「Aで22/22に到達できるか」（NanoLang等）は再現性にブレが出やすい。
- **Vera/A の status 再分類**：上述のとおり SKIPPED→PARTIAL(0/22) に補正した。生の自己申告は `results/Vera/result.json` の `reclassified_note` に保持。
- **`self_recovered` の解釈統一**：エージェントの生値は「修正で回復したか」を返し pass@1 時に false としていたため、レポートでは「人間介入なしに22/22到達」で正規化した（生値も result.json に併記）。
- **ツールチェーンのバージョン差・ドリフト**：MoonBit は公式インストーラ取得のためビルド時点の最新（2026-05-29ビルド）。`?` 演算子廃止などの言語変化が修正ループに影響した。MoonBit の `llms.txt` は 404 で、ガイド取得は `moon` のヘルプ／スキャフォールド読解にフォールバックした。Zero のスキルIDも公式ガイド記載の `zero-language` ではなく実際は `language` だった（取得時に補正）。
- **NanoLang はインタプリタ実行**：イメージには C トランスパイラ（`nanoc`）ではなく同梱インタプリタ（`nanolang`）のみが入っており、ガイド記載の「Cにトランスパイル」経路ではなくツリーウォーク実行で採点した。
- **採点ハーネスは各エージェント自作**：ただし全処理系はコンテナ内の固定版で動かし、22/22主張は全てレポート側で独立再実行して確認済み（上表）。「期待値の解釈」（例：`3.14`を小数として一致とみなす等）は仕様の読み筋に依存する。
- **再現範囲**：Docker で再現できるのは「生成済みパーサ × 22ケースの採点」。コード**生成**自体はAIエージェント駆動でコンテナ外（`AGENT_INSTRUCTIONS.md` 準拠）。詳細は `DOCKER.md`。
