# 実行指示書：AI生成の質ベンチマーク（問題1：JSONパーサ）

あなたはこのベンチマークを **一気通貫で自動実行する** エージェントです。
6つの言語それぞれに対して「コード生成 → 実行 → 失敗したら修正ループ → 採点」を行い、
最後に比較レポート（`REPORT.md`）を1つ出力してください。

途中で人間に確認を求めず、最後まで自走してください。
ただし後述の「安全上の制約」は必ず守ること。

---

## 対象言語（この順で処理する）

1. JavaScript
2. Python
3. Zero (zerolang)
4. MoonBit
5. NanoLang
6. Vera

---

## 全体の流れ

各言語について、以下を順番に行う。

### ステップ0：処理系の確認
- その言語の処理系（コンパイラ/インタプリタ）がこの環境で使えるか確認する。
- **使えない場合は、その言語を `SKIPPED` として記録し、理由（インストール不可／コマンドが無い等）を残して次の言語へ進む。** 無理にインストールを試みて環境を壊さないこと。
- 使える場合のみ次のステップへ。

### ステップ0.5：文法ガイドの取得（新言語のみ・パターンB用）
- Zero / MoonBit / NanoLang / Vera は、リポジトリ直下の `guides/{言語名}.md` に
  **公式ガイドの取得方法**が書いてある。その手順で公式ガイドを取得する。
  （例：Vera は `curl -fsSL https://veralang.dev/SKILL.md`、
   Zero は `zero skills get language`）
- 取得した公式ガイド本文を、その言語の `prompts/{言語名}.md` のパターンB に差し込む。
- JavaScript / Python はガイド不要（パターンB はスキップ）。

### 生成は2条件で行う（新言語のみ）
- **パターンA**：仕様のみで生成（モデルが言語をどれだけ知っているか）
- **パターンB**：公式ガイドを与えて生成（設計で訓練データ不足を補えるか）
- JavaScript / Python はパターンAのみ。
- それぞれの結果を `results/{言語名}/A_*` と `results/{言語名}/B_*` に分けて保存し、
  result.json にも A / B 両方の pass@1・修正回数を記録する。

### ステップ1：コード生成
- `spec.md`（このディレクトリにある仕様）を読み込む。
- `prompts/{言語名}.md` にプロンプト雛形がある。それに仕様を埋めて、自分自身で当該言語のJSONパーサを実装する。
- 生成したコードを `results/{言語名}/attempt_1.{拡張子}` に保存する。
- このとき **生成コードの行数と概算トークン数（単語数でよい）を記録** する。

### ステップ2：テスト実行
- `testcases.md` にある全テストケースを、生成したコードに対して実行する。
- 実行方法は各言語に合わせてよい（テストランナーを書く、main関数で回す等）。
- 各ケースが「期待通りか」を判定する。

### ステップ3：修正ループ（失敗時のみ）
- 1つでも失敗したら、**その言語の処理系が出した生のエラー出力／失敗内容だけ** を根拠に修正する。
- 人間向けの解説を自分で足さず、生エラーを見て直すこと（エラー出力の質を測るため）。
- 修正したコードを `attempt_2`, `attempt_3`... と保存する。
- **修正は最大5回まで（＝最大で `attempt_6` まで）。`attempt_6` でも全ケース通過しなければ、その時点で打ち切り、status は `PARTIAL`、その attempt の通過数を `final_pass_count` とする。** 5回に満たないうちに諦めて打ち切ってはならない（未達なら必ず5回まで修正を試みる）。**5回を超えて修正した結果は採用しない**（規定外成功は PARTIAL 扱い）。
- **`fix_rounds` は「全ケース通過に到達した attempt 番号 − 1」で数える**（例：`attempt_1` で通過＝0、`attempt_3` で通過＝2）。未達のまま打ち切った場合は `fix_rounds = 5`。**`fix_rounds` と最終 attempt 番号は必ず整合させること**（食い違いを残さない）。
- 各 attempt で「何ケース通ったか」を記録する。

### ステップ4：記録
- その言語の結果を `results/{言語名}/result.json` に保存する（後述のフォーマット）。

## 固定ルール（公平性・再現性のため事前に明文化）

このベンチの結論を「同一条件の比較」として成立させるため、次を全条件で固定する。

1. **手書き必須・標準JSON禁止（全条件）**：`JSON.parse`／`json.loads`／`eval` 等の標準ライブラリJSON機能は使用禁止。再帰下降などで自分で実装する（プロンプト雛形の制約にも明記済み）。
2. **パターンA で参照してよい情報**：①仕様、②処理系の起動方法（`--help`・コンパイル/実行コマンド）、③**自分の生成コードをコンパイル/実行して出た生のエラー・診断出力**のみ。
   - **禁止**：文法ガイド、エラーコードの解説コマンド（例 `zero explain`）やその例示、標準ライブラリ解説、サンプルプログラム、Web情報。要するに「自分のコードに対する生エラー」を超える“教材的情報”は A では使わない。
3. **パターンB で渡すガイド**：`guides/{言語名}.md` の手順で取得した**公式ガイド本文**のみ。取得元・コマンドを `toolchain_notes` に必ず記録する。
4. **合格判定（pass criterion）の統一**：
   - 正常系13ケースは「パース結果の**値**が期待と一致」、異常系9ケースは「**異常終了せずエラー**として扱う」で合否を出す。
   - 比較は**ネイティブな値の等価**を原則とする。値ADTを構築できない言語が**正規化シリアライズ（正準形の文字列/バイト比較）**で代替した場合は合格扱いにしてよいが、`result.json` の `pass_basis` に **`value`（値比較）/ `normalized`（正規形比較）** を記録し、サマリー表でも別カテゴリとして示す。
5. **期待値ハーネスの固定**：22ケースの入力と期待は `testcases.md` を唯一の正とし、全条件同一。各エージェントが書くのは“走らせ方”だけで、期待値は変更しない。最終確認はレポート作成側がコンテナ内で**独立再実行（採点の二重化）**する。
6. **修正回数ルールの厳守**：上記ステップ3の通り（最大5回＝`attempt_6` まで、未達は `PARTIAL`、`fix_rounds = 到達attempt − 1`、未達は5、規定外成功は不採用）。

## 採点基準

各言語について以下を記録する。

| 指標 | 説明 |
| --- | --- |
| status | `DONE` / `PARTIAL`（5回でも全通過せず）/ `SKIPPED`（処理系なし） |
| pass@1 | 最初の生成（attempt_1）で全ケース通過したか（true/false） |
| fix_rounds | 全ケース通過までの修正回数（＝到達attempt − 1、0〜5。未達は5。最終attempt番号と整合させる） |
| final_pass_count | 最終的に通ったケース数 / 全ケース数 |
| loc | 生成コードの行数（最終版） |
| tokens | 概算トークン数（最終版、単語数でよい） |
| self_recovered | 人間の介入なしに自力で修正しきれたか（true/false） |
| error_quality | エラー出力が修正に役立ったかの所感（free / structured / poor など短く） |
| pass_basis | 合否判定の基盤：`value`（ネイティブ値の比較）/ `normalized`（正規化シリアライズの比較） |

---

## result.json のフォーマット

```json
{
  "language": "Python",
  "status": "DONE",
  "pass_at_1": true,
  "fix_rounds": 0,
  "final_pass_count": "12/12",
  "loc": 45,
  "tokens": 210,
  "self_recovered": true,
  "error_quality": "free-text but clear",
  "pass_basis": "value",
  "notes": "標準ライブラリを使わず手書きで実装。境界条件も一発で通過。"
}
```

SKIPPED の場合：

```json
{
  "language": "Vera",
  "status": "SKIPPED",
  "reason": "処理系がこの環境にインストールされていない"
}
```

## 最終レポート（REPORT.md）に含める内容

> **必読**：REPORT.md の書式・検証ルールは全 problem 共通の規約 [`docs/REPORT_GUIDELINES.md`](../docs/REPORT_GUIDELINES.md) に従うこと（数値主張は再実行コマンド併記＋独立再実行で確認、因果は判別シグナル付き、用語は日常語＋原語、status / self_recovered の扱い 等）。

1. **概要**：使用したモデル名、実行日、何言語中何言語が実行できたか
2. **結果サマリー表**：全言語 × 上記指標の一覧表
3. **言語ごとの所感**：各言語2〜4行。生成のしやすさ、つまずいた点、エラーの質
4. **考察**：
   - pass@1 が高かった言語と低かった言語の差は何に由来するか
   - 訓練データが豊富な既存言語（JS/Python）と新言語の差
   - 構造化エラーを持つ言語（Zero等）が修正ループで有利だったか
5. **Caveats**：SKIPPEDになった言語、試行回数の少なさ、結果のブレなど正直に書く

---

## 安全上の制約（必ず守る）

- システム設定の変更、sudo を伴う大規模インストール、外部への送信は行わない。
- 処理系が無い言語は **SKIPPED にするだけ** とし、無理にビルドして環境を壊さない。
- 生成・実行するコードはこのベンチマークの課題に限る。
- 不明点があっても止まらず、判断を `REPORT.md` の Caveats に記録して続行する。
