# 共通採点基準（第三者オラクル）— 問題2 銀行口座

このファイルは、各エージェントの**自作ハーネスとは独立に**、生成コードを `spec.md` /
`testcases.md` に照らして採点するための明文化された基準である。
採点はコードの**ソースを読んで**判定する（自作ハーネスの「PASS N/14」表示を信用しない）。

> 背景：各言語のハーネスはエージェントが自作している。ハーネスは自分の実装と整合的に
> 書かれているため、実装が仕様を弱く満たしていてもハーネス上は「PASS」になりうる。
> 特に異常系で顕在化する（[後述](#異常系の拒否要件)）。

---

## 1. 「エラー / 拒否」と認める条件（最重要）

`testcases.md` の異常系は **「ERR＝操作が拒否され、残高が変わらない」** を要求する。
これは2条件の AND である：

- **(a) 拒否シグナル**：呼び出し側がコード上で**成功と失敗を判別できる**こと。
- **(b) 残高不変**：拒否された操作の前後で残高が変化しないこと。

(a) の「拒否シグナル」として**認める**のは、次のいずれか：

| 区分 | 形 | 例 |
| --- | --- | --- |
| ✅ 例外 | 型付き例外を送出（呼び出し側が catch して判別） | JS `throw new Error`, Python `raise BankAccountError`, MoonBit `raise BankError` |
| ✅ 直和型 | `Result`/`Either`/`Outcome` 等の **Err ヴァリアント**を返す | NanoLang/B `Result.Err`, Vera `Err`/`Result<_,String>` |
| △ 番兵値 | 正常値の**定義域外**にある明確な失敗値を返し、呼び出し側が実際に判定する | Zero `-1`（残高は常に >= 0 なので -1 は衝突しない） |

(a) として**認めない**のは：

| 区分 | 形 | なぜ不可 |
| --- | --- | --- |
| ❌ 無シグナル | 不正操作時に**正常時と同じ型・同じ定義域の値**（例：変更前の残高）を返し、差異が標準出力への print のみ | 呼び出し側が「拒否された無操作」と「成功した無操作」を**値で区別できない**。print は戻り値ではないので分岐に使えない |

> △ 番兵値は ✅ より弱い（型システムが取り違えを防がない）。採点上は通過とするが、
> REPORT では「拒否シグナルの強さ」の列で区別する。

### 異常系の拒否要件

異常系ケース **7〜12** は、ハーネスが **(a) 拒否シグナルを実際に検査している**ことを
必須とする。残高 (b) だけを見て「変わっていないから PASS」とするハーネスは、
そのケースを**通過とみなさない**（拒否が立証されていないため）。

不変条件ケース **13・14** は残高の不変性そのものを問うので、(b) のみで判定してよい。

## 2. 正常系（ケース 1〜6）

- 操作後の残高が期待値と一致すれば通過。エラー扱いの設計差は問わない。

---

## 3. 採点の単位と表記

- `final_pass_count` は `N/14`。各ケースは上記基準で**個別に**通過/不通過を判定。
- `status`：`DONE`（14/14）／`PARTIAL`（一部未達）／`SKIPPED`（処理系なし）。
- 番兵値による拒否は通過とするが、`error_channel` を `sentinel(weak)` と注記する。
- 例外・直和型は `exception` / `sum-type`。無シグナルは `none`。

---

## 4. 検証手順（再現可能性）

1. 各条件の**最終 attempt** を、コンテナ内固定処理系で実行する。
2. **コマンド・stdout・stderr・終了コード**を `results/<lang>/<cond>/rerun.log` に保存する
   （`scripts/rerun_problem2.sh` が一括取得）。
3. ただし「PASS N/14」表示は**補助証拠**にすぎない。最終判定は本ルーブリックを
   ソースに適用した第三者採点（`results/REGRADE.md`）で行う。
