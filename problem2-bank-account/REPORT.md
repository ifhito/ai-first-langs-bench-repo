# ベンチマークレポート：問題2 銀行口座（全6言語・Docker検証）

## 概要

- **使用モデル**: Claude Opus 4.8 (1M context) — `claude-opus-4-8[1m]`
- **実行日**: 2026-06-07
- **実行体制**: 言語×条件ごとに**独立したサブエージェント**（コンテキスト分離）を割り当て、計10条件を並行実行。
  - パターンA は「仕様＋処理系の起動方法」のみを与え、**言語の文法は教えない**（モデルの素の知識か、処理系のエラーからのみ復元させる）。A と B を別エージェントにして、A の測定に B のガイド知識が混ざらないようにした。
- **実行環境**: 再現可能な Docker イメージ `ai-lang-bench`（全処理系をバージョン固定で同梱）
  - JavaScript: Node v22.22.3 ／ Python: 3.13.5 ／ Zero: 0.2.1 ／ MoonBit: 0.1.20260529（3e1c753 2026-05-29）／ NanoLang: 同梱インタプリタ（`main` ビルド）／ Vera: 0.0.160（Z3 同梱）
  - 生成コードはコンテナ内の固定処理系でコンパイル・実行し、**14ケース**を採点
- **対象**: 6言語中6言語が実行可能（処理系不在による SKIPPED は無し）
  - JavaScript / Python … パターンA のみ（訓練データ豊富な対照群）
  - Zero / MoonBit / NanoLang / Vera … パターンA（仕様のみ）＋パターンB（公式ガイド付き）の2条件
- **到達**: 第三者再採点（[`RUBRIC.md`](RUBRIC.md) / [`results/REGRADE.md`](results/REGRADE.md)）の結果、**10条件すべてが 14/14（全 DONE）**。各条件の合否は自作ハーネスの「PASS N/14」表示ではなく、**異常系で拒否シグナルを呼び出し側が判別できるか**をソースで確認して判定している。

### この問題のいちばん大事な発見

14ケースは「ロジックの正しさ」よりも **「不変条件 `残高 ≥ 0` を誰が保証するか」** を問う。
全条件が 14/14 に到達したため、差は **pass/fail ではなく「不変条件の保証のされ方」とコスト**に出た。

- **言語そのものが不変条件（残高 ≥ 0）を機械的に保証したのは Vera/B（ガイドあり）ただ1条件**。
  残高を **「0以上であることが型の時点で保証される型」**で表し、専用の検証コマンド（`vera verify`）が、その「常に0以上」を含む **42個の条件を数学的に証明**した（ただし `deposit`/`withdraw` の事後条件自体は実行可能版では `ensures(true)`。保証の主体は `Nat` 型と検証器であり、手書き契約ではない。[Caveats](#caveats正直な注意書き) 参照）。
- 残りの9条件はすべて、**AI が自発的に書いた防御コード**（`if amount <= 0 then エラー`、ミューテーション前のガード等）＋**明示的なエラーチャネル**（例外・直和型・番兵値）が支えで、言語・型は不変条件そのものを止めてくれない。
- **「拒否を呼び出し側が値で判別できるか（エラーチャネル）」が隠れた第2の軸**。異常系（#7〜#12）は、エラーを例外/直和型/番兵値のいずれかで返さないと「拒否された無操作」と「成功した無操作」を区別できない。本採点はこの判別可能性を必須要件にしている（[`RUBRIC.md`](RUBRIC.md)）。

## 用語の説明

このレポートと `results/{言語名}/result.json` で使う用語の定義。

### 生成条件：パターンA / パターンB
同じ仕様・同じ14ケースで、**与える材料だけを変えて**比較する。

| 条件 | 与えるもの | 測りたいこと |
| --- | --- | --- |
| **パターンA（仕様のみ）** | 銀行口座の仕様だけ。言語の文法ガイドは**与えない**（処理系の呼び出し方などツール操作の確認は可） | モデルが**その言語を訓練データからどれだけ知っているか**（=素の言語知識） |
| **パターンB（ガイド付き）** | 仕様に加えて、**公式の文法ガイド**（`zero skills get`／`veralang.dev/SKILL.md`／NanoLang同梱docs 等）を取得して与える | **設計介入（ガイド供給）で訓練データ不足を補えるか** |

- JavaScript / Python は訓練データが豊富な対照群のため**パターンAのみ**。
- Zero / MoonBit / NanoLang / Vera は**A・B両方**を実施し、`results/{言語}/result.json` の `conditions.A` / `conditions.B` に別々に記録している。

### attempt（試行）と修正ループ
1回の生成で全ケース通るとは限らない。失敗したら**処理系の生エラーだけ**を根拠に直し、再実行する。この各回が **attempt**。
- `attempt_1.<ext>` = 最初の生成。`attempt_2` … = 各修正後のコード（全部残す）。
- **修正は最大5回まで**（`fix_rounds` が修正回数）。5回で全通過しなければ打ち切り `PARTIAL`。
- 修正時は人間向けの解説を足さず、**生エラーを見て直す**（その言語のエラー出力の質を測るため）。

### その他の指標
- **pass@1（1回でpass）**：最初の生成だけで全14ケース通ったか。
- **final_pass_count（最終）**：最終的に通ったケース数（`N/14`）。
- **status**：`DONE`（14/14）／`PARTIAL`（5回でも未達）／`SKIPPED`（処理系が無い）。
- **self_recovered（自己解決）**：人間の介入なしに 14/14 へ到達したか（pass@1 も ✅ に正規化。生値は result.json 併記）。
- **loc / tokens（最終行数／トークン数）**：最終版コードの行数／概算トークン数（単語数）。
- **error_quality（エラーの質）**：◎ とても親切／○ ふつう／△ 役に立たない（表直後に定義）。

## 検証（採点の独立再採点と再実行ログ）

検証は**2トラック**で行った。両者を分けるのは、**各エージェントの採点ハーネスは自作**であり、

**トラック1：再実行ログの保存（証拠）.**
全10条件をコンテナ内固定処理系（JS/Python はローカル）で再実行し、
**コマンド・stdout・stderr・終了コードを `results/<lang>/<cond>/rerun.log` に保存**した

[`RUBRIC.md`](RUBRIC.md) を各条件の最終 attempt の**ソースに適用**し、特に異常系で
**拒否シグナルを呼び出し側が判別できるか**を確認した（[`results/REGRADE.md`](results/REGRADE.md)）。
ハーネスの「PASS N/14」表示は補助証拠にすぎない。

## 言語ごとの所感

### JavaScript（A）
仕様だけでクラス（`deposit`/`withdraw`/`balance`＋`throw`）を一発生成、14/14。`#balance` を私有フィールドにし、`#assertPositiveInteger`（`Number.isInteger`＋`<=0`）と残高超過ガードを**ミューテーション前**に置く防御コードを自発的に記述。**言語は不変条件を止めてくれない**ため、保証は手書きチェック頼み。

### Python（A）
同じく一発 14/14。独自例外 `BankAccountError`＋`_validate_amount`（非整数・bool・0以下を拒否）と残高超過チェックを検証後にだけ代入。**不変条件は完全に AI の手書きチェック**で、言語の強制はゼロ。最短35行。

### Zero（A→14/14・修正1回／B→14/14・修正4回）
**「型はあるのに、この環境では使えない」逆説**が両条件で出た。構造体やタグ付きの型を使った“本格的”な書き方は型検査は通るのに、実際に実行ファイルを作る段階でエンジンが対応しておらず**動かせない**。エラーが「どの構文が動かせないか」を毎回具体的に名指ししてくれるので、それに従って**型を使わず整数だけで組み直し**、ようやく実行できた。エラーが具体的なおかげで、ガイド無しの A でも到達できた。不変条件の保証は手書きチェック頼み。

### MoonBit（A→14/14 pass@1／B→14/14・修正2回）
**A は一発 14/14**。ML系で成熟しモデルが熟知。型の仕組みで「エラーを無視できない」ようには強制されるが、整数型に「0以上」を表す仕組みが無く、**`残高 ≥ 0` 自体は手書きチェック**。B は公式の配布ドキュメントが取得できなかったため、ツールチェーン自体（ひな形生成やテストの診断メッセージ）をガイド代わりにした。ラベル付き引数の書き方を1か所間違えたが、エラーが正しい書き方を添えて示したので1回で直せた。

### NanoLang（A→14/14 pass@1／B→14/14・修正1回）
**A は 14/14**。不変束縛（再代入は `set`）に合わせ口座を plain int で表現しつつ、異常系の拒否を値で返すために **直和型 `union OpResult { Ok{balance:int} ｜ Err{code:int} }`** を導入。`deposit`/`withdraw` は `OpResult` を返し、**Err ヴァリアントそのものが拒否シグナル**になるので、呼び出し側は `is_err()`/`match` でコード上分岐できる（print ではない）。異常系 #7〜#12 は「`is_err`==1（拒否）AND 残高不変」を AND で検査。決め手は処理系エラー `Expected '{' after variant name (got LPAREN)` で、「variant は `{ field: type }` で payload を持てる」と気づいて直和型に到達した（**公式ガイドは読まず処理系エラーのみ**から発見）。本採点者がソース確認＋コンテナ2回再実行（`TOTAL_PASSED 14` 決定的・exit=0）で検証。なお A では `shadow` テストを書かず、インタプリタは「missing shadow test」を**警告止まり**で実行した（＝言語が強制するわけではない）。

B は同梱docsを読み **`Result<Account,string>`（generic 直和型、`Ok` に `Account` 構造体を包む）** 設計で到達し、`is_error` で拒否を判定して **14/14**（stdlib import パス不解決を1回修正）。A は generic 化やネスト構造体 payload が型検査を通らない壁に当たり primitive int payload の `OpResult` に寄せた独立設計で、**ガイド無しでも直和型エラーチャネルに到達できる**ことを示した。A/B 差は「ガイドが必須」ではなく**ガイドが設計到達を安定させる**方向に出た。

### Vera（A→14/14・修正4回／B→14/14・修正1回）
**この課題の主役**。A はガイド無しでも、エラーが「契約の書き方の見本」や「使える番号の範囲」を具体的に示してくれたおかげで、Vera 独特の書き方（値を名前ではなく番号で指す方式と、関数ごとに付ける契約宣言）を自力で組み上げ 14/14。ただし保証は手書きの構造（エラー時に元の残高を返す形）で、契約を検証器に証明させるところまでは A では届かなかった。
B は公式ガイドを読み、**残高を「0以上であることが型の時点で保証される型」**で表現。専用の検証コマンドが、その「常に0以上」を含む **42個の条件を数学的に証明**した。**不変条件を「言語が保証する性質」にできた唯一の条件**。

## 考察

### pass@1 の差はどこから来るか（因果は判別シグナル付き＝規約B-5）
- **訓練データ量が pass@1 を左右した可能性が高い**（n=1 のため断定はできない）。JS/Python と **MoonBit/A** は A で即 pass@1（→「①モデルが元から知っていた」寄り）。
  - **NanoLang/A** も rubric 上 pass@1 で 14/14 だが、直和型の文法は処理系エラーからの probe で発見しており「①素の知識」と「②エラー駆動の探索」の中間。文法の細部（payload-union の書き方）は知識に無く、エラーが埋めた。
- **A=失敗だが最終全通過**の Zero（A/B）・Vera（A/B）・MoonBit/B・NanoLang/B は「②設計・親切なエラーで通った」寄り：
  - **Vera/A**：pass@1 ❌ → 4回修正で 14/14。エラーが「契約の書き方の見本」「使える番号の範囲」を具体的に示すので、未知の文法をエラーだけから組み上げられた（エラーが事実上ガイド代わり）。
  - **Zero/A・B**：エラーが「動かせない構文」を毎回名指しし、型を使う設計から整数だけの設計への切り替えを後押しした。

### この問題の核心：不変条件は「言語が保証」したか「AIが防御コードで」担保したか
14ケース（特に #7〜#12）は、明示チェックや**エラーチャネル**を書き忘れれば簡単に「成功」として通ってしまう。各条件の `notes` と再採点を集約すると：

| 担保の主体 | 条件 |
| --- | --- |
| **言語/型/検証器が機械的に保証**（残高 ≥ 0） | **Vera/B のみ**（残高を `Nat` 型で表し、`vera verify` が42条件をZ3証明。`deposit`/`withdraw` 契約自体は実行版で `ensures(true)`、保証の主体は型＋検証器） |
| **AIの手書き防御コード＋明示エラーチャネル**（言語は強制せず）で 14/14 | 残り9条件すべて（JS・Python・Zero A/B・MoonBit A/B・NanoLang A/B・Vera/A） |

- **全条件が 14/14 に到達したので、pass率は判別力を失う**。差が出たのは「**保証のされ方**」：同じ 14/14 でも、Vera/B だけが「型と検証器が `残高 ≥ 0` を**証明**」しており、他は「テストが**たまたま**通っている（次のケースを足せば抜けるかもしれない）」状態。
- **エラーチャネルは全条件が備えている**（例外・直和型・番兵値）。これは「異常系を呼び出し側が判別できる」最低条件で、本採点が必須にしている。番兵値（Zero）は通過扱いだが型安全性では弱い（[下記](#コスト行数トークンの見方規約b-6) / `RUBRIC.md`）。
- **MoonBit/Zero の型システムは部分的にしか効かない**：「エラーを無視させない」ところまでは型で強制できるが、**「0以上」のような数値の範囲を表す型が無い**ため `残高 ≥ 0` 本体は手書きに落ちる。
- **Zero は「型があるのに実行できない」二重の罠**：型を使った版は型検査は通るのに、実行ファイルを作る段階でエンジンが対応せず動かせない。結局は整数だけの版に退化し、型の安全性は採点に寄与しなかった。
- **Vera/A と Vera/B の対比が示すもの**：同じ Vera でも、ガイド無し（A）では契約の Z3 証明に届かず手書き構造で通したのに対し、ガイドあり（B）では `Nat` 型＋契約で**機械的保証**に到達した。「不変条件を言語に肩代わりさせる」には**公式ガイドという設計介入が要った**。

### コスト（行数・トークン）の見方（規約B-6）
- pass率が同じ 14/14 でも中身は大きく違う。**MoonBit は最短（27〜28行）**で型を活かして簡潔。**NanoLang/B（245行）・Zero/B（227行）は冗長**：Zero/B は型が使えず純粋ヘルパ＋番兵値へ退避したため肥大。pass率だけ見ると「設計で苦労して通した」ケースを過大評価するため、行数・トークンを併記している。

### 構造化エラーは修正ループで有利だったか
- **強く支持された**。最良の実例は **Vera/A**：公式ガイド無しで、「契約の書き方の見本」と「使える番号の範囲」を示す具体的なエラー**だけ**から、未知の文法を組み上げて 14/14。Zero の「動かせない構文の名指し」、NanoLang の「代入は set と書く」というヒント、MoonBit の「正しいラベル記法の提示」も修正を直接後押しした。
- **ただし構造化エラーが効くのは「文法復元」まで**：NanoLang/A は `Expected '{' after variant name` から直和型の**書き方**を復元できたが、コンパイラは「エラーチャネルを設けよ／拒否を値で返せ」とは指摘しない（型が要求しないため、print で済ませることも文法上は通る）。エラーチャネルを**設計判断として選んだ**のはモデル側で、言語の強制ではない。構造化エラーは文法に効き、設計上の不変条件は依然 AI 任せ、という切り分け。
- **限界**：Vera では「引数の受け取り方」だけがエラー上で文法内部の符号として表示され（実際に書くべき記号が分からない）、ここだけは復元しづらかった。「見た目の親切さ」より「**足りない知識をズバリ指せるか**」が効く。

## Caveats（正直な注意書き）

- **試行回数は各条件1回（n=1）**：LLM生成にはブレがあり、別シードでは pass@1・修正回数・最終行数が変動しうる。特に「A で 14/14 に到達できるか」「Vera/A が契約のZ3証明まで届くか」は再現性に幅が出やすい。
- **「言語が保証 = Vera/B のみ」は本実行での観測**：他言語でも、より凝った設計（独自の検証ステップ等）を取れば機械的保証に近づける可能性はある。本ベンチは「仕様＋（ガイド）で素直に書いたとき何が起きたか」を測っている。
- **Vera の検証器のクセ**：`vera verify` は match 本体の関数に対する `ensures` で偽の反例を出すことがあり（Vera/A）、Vera/B でも runtime-check codegen が match を含む事後条件を降ろせず（CodegenSkip）、実行可能版では一部の事後条件を `ensures(true)` に緩めた。ただし `Nat` 型による非負保証と42条件のZ3証明は実行可能版で成立している（`results/Vera/result.json` に詳細）。
- **Zero は「型検査」と「実行可能性」が別物**：型検査が通っても、実行ファイルを作る段階で対応しておらず動かせない構文が多い。本ベンチは**実行して14ケースを通すこと**を採点基準にしたため、型付き版でなく実行可能版を最終とした。Zero/A は画面出力の方法を復元できなかったため、通過数を**プログラムの終了コード**で表現している（終了コード=14）。
- **NanoLang/A は `shadow` テスト無し**：インタプリタは「missing shadow test」を警告止まりで実行する（テストは言語仕様上は推奨だが、実行時に強制されない）。
- **Python の `run_tests.py` は終了コードで失敗を返さない**：`sys.exit()` が無く、失敗時も exit=0。本レポートの合否は終了コードではなく **stdout の「PASS 14/14」表示**に依拠している。Zero/A のように終了コード=通過数で表す方式の方が機械検証には堅い。
- **ガイド取得のばらつき**：MoonBit は公式配布ドキュメントが取得できなかったため、ツールチェーン自体をガイド代用。Zero はガイド取得コマンドの名称が公式表記と一部違った。NanoLang は同梱ドキュメントが一次情報。
- **「採点の二重化」は独立オラクルではない**：各エージェントの採点ハーネスは自作であり、その「再実行」は同じ自己採点器をもう一度動かしただけ。自作ハーネスは実装と整合的に書かれるため、異常系で残高しか見ないと「拒否を返さない実装」も PASS になりうる。そこで本リビジョンでは、(1) コマンド・stdout・stderr・終了コードを `results/<lang>/<cond>/rerun.log` に保存し、(2) `RUBRIC.md` を**ソースに適用した第三者再採点**（`results/REGRADE.md`）を判定の基準にした。なお番兵値（Zero）は通過扱いだが型安全性では例外・直和型より弱い（`sentinel(weak)` と注記）。今後はエージェント横断の**固定共通ハーネス**にするのが望ましい。
- **再現範囲**：Docker で再現できるのは「生成済みコード × 14ケースの採点」と「Vera/B の契約Z3検証」。コード**生成**自体はAIエージェント駆動でコンテナ外（`AGENT_INSTRUCTIONS.md` 準拠）。詳細は `DOCKER.md`。
